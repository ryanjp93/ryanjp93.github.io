Design Specification
 The client for this website is a Twitter user and blogger with 1500 followers and reader numbers ranging between 300 and 2000 depending on the type of blog post. They requested a site to replace their existing WordPress one which had a number of shortcomings such as a bad mobile version and a very simplistic overall design. Although I was given free reign on design, a set of requirements were produced through questioning and taking the client's audience statistics into account:

- Mobile/responsive focus: around 65% of the user base are on mobile
- Performant: mobile users tend to have performance constraints and slower speeds
- Direct linking: necessary to promote posts on twitter
- Searching: the client expects to eventually have a large number of posts
- Twitter feed: Twitter is the foundation of the user base and so it should be interlinked
- Analytics and user statistics: The WordPress site had this, to not have it would be a downgrade
- Ads: the client is working on increasing their user base, ads would cover hosting costs

In terms of visuals, I decided that an edgy, clean look would complement the client's modern, stat-focused blogging style. I also limited the colour range to three main colours that I felt fit this theme. Due to the user base being Twitter focused, I also took two elements of its design and incorporated them into the site. Firstly, the top bar which fixes itself to the top of the screen as the page is scrolled and secondly, the fixed background on the client's Twitter feed. This was also worked into the lock screen style landing page that the site now has.

Site Responsiveness
 Like the other sites showcased on this portfolio, this site utilises a number of responsive design traits such as media query breakpoints and table/flexbox styling. Building around mobile proved to be a good decision, as the site's main content styling remains essentially unchanged across all devices. The Twitter feed is hidden on devices smaller than an iPad and the ads are shuffled around to different places instead. To better accommodate phone users, the main menu bar shown on the landing page also snaps to the top of the screen when the user scrolls past it. Lazy Loading Adverts and Twitter
 A slightly unusual complexity in this site lies in the page ads, as ads typically do not play nice with media breakpoints. The ads themselves do not resize and are always loaded at a fixed size. This would mean that desktop ads have little chance of fitting on a mobile screen and mobile ads would look very out of place on a desktop screen. Loading in eight different ads (two for mobile, two for large mobile, two for tablet and two for desktop) and simply hiding or unhiding them based on the browser window size would not be acceptable as it would slow down the initial loading time for mobile users. I therefore decided that a lazy loading approach would be best- load only the two relevant ads and listen for resize events to swap in new ones. While this was easy to implement for the Twitter feed (as it does not show on small devices), it was surprisingly difficult for the ads as some of them use a document.write call. Document.write is a function that will only ever fire before a page is loaded, any use after that point will be ignored. Given that I wanted the adds to load as requested, this meant I had to partially rewrite the ad scripts.

Fixed Background Image
 While visually impressive, the fixed background brought its fair share of problems. On iOS there is a bug in the way background-size: cover is handled when used with background-attachment: fixed to do with the way it reads the size of the document body. This bug causes the background to become stretched out and unrecognisable. To fix it, I put the background on a pseudo-element attached to the document body itself and fixing its position. While this solved the bug, I was not happy with the scroll lag it seemed to have generated. I eventually discovered two reasons for this, the first being an unrelated scroll event listener which was constantly changing a CSS property, and the second being that when a fixed background is scrolled, it causes a repaint on every scroll event. After some research, I found that setting will-change: transform on the element tells the browser to allocate it its own layer, reducing the impact of constant paints.

Performance and Optimization
 In addition to the work on reducing the number of ad requests and the number of scroll paints, I used a number of other small optimizations. Whenever a request is made for a post or image, they are cached temporarily in case the user navigates to a page or runs a search which includes the same post or image. At the point the page is loaded, it checks this cache before making any requests and can make loading times appear instant, even on mobile. Fewer requests also means a lower data impact of using the site for mobile users. The final and most simple optimization was to ensure that all of thee site assets were web optimized and minified.
